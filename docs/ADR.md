# Architecture Decision Records

## 철학
실거래로 실제 돈이 오가는 시스템이다. **안전(리스크 차단)과 데이터 무결성이 속도보다 우선**한다.
모든 자동 주문은 알고리즘(규칙) → LLM(OpenAI, 판단) → 리스크 게이트(차단)의 다단계 승인을 통과해야 한다.
SDD → TDD를 강제해 매매 로직의 회귀를 막는다. 외부 의존성은 backend에 격리해 프론트가 직접 거래소/LLM을 호출하지 못하게 한다.

> 네이밍 주의: 아래 ADR-001/002/005는 초기 설계 시점 기록이라 본문에 "Claude"가 남아 있다.
> 실제 LLM 제공자는 **OpenAI**다(ADR-007이 대체). ADR은 결정 이력이라 과거 본문은 보존하고
> 대체 관계만 표시한다.

---

### ADR-001: 프론트/백 분리 (Next.js + FastAPI)
**결정**: 프론트엔드는 Next.js 14, 백엔드는 별도 FastAPI 서버로 분리한다.
**이유**: 알고리즘·에이전트가 Python(Pandas, TA-lib) 생태계에 묶여 있고, Robinhood MCP·Claude 호출을 한 곳(backend)에 격리해야 보안·리스크 통제가 가능하다. 실시간 push는 FastAPI WebSocket으로 처리.
**트레이드오프**: 단일 Next.js 풀스택 대비 배포 단위·타입 동기화 비용 증가. 타입은 수동 동기화로 감수.

### ADR-002: 알고리즘은 순수 함수, 에이전트는 상태 루프
**결정**: `algorithms/`는 부수효과 없는 순수 함수(입력 DataFrame→출력 시그널/사이즈), `agents/`는 주기 실행 루프로 분리한다.
**이유**: 매매 핵심 로직(시그널/사이징)을 외부 의존성 없이 단위 테스트(TDD)할 수 있어야 한다. I/O(MCP, Claude, DB)는 에이전트 레이어로 밀어낸다.
**트레이드오프**: 레이어 경계 유지 비용. 그러나 테스트 용이성과 회귀 방지 이득이 크다.

### ADR-003: 리스크 에이전트 kill-switch는 PreToolUse hook으로 강제
**결정**: 주문 실행 툴 호출 전 PreToolUse hook이 리스크 에이전트를 강제 호출하고, 한도 초과 시 주문을 차단한다.
**이유**: 코드 경로 어디서 주문이 나가든 단일 게이트를 통과하게 만들어, 버그·우회로 인한 한도 초과 주문을 원천 차단한다. 안전이 최우선.
**트레이드오프**: 모든 주문에 리스크 체크 지연이 추가된다. 안전을 위해 감수한다.

### ADR-004: 개발 DB는 SQLite, 캐시/실시간은 Redis
**결정**: 개발은 SQLite로 시작하고 프로덕션에서 PostgreSQL로 전환한다. 실시간 가격·에이전트 공유 상태는 Redis.
**이유**: 1인 개발 초기 속도를 위해 SQLite. SQLAlchemy로 추상화해 전환 비용을 낮춘다. 1초 갱신 실시간 데이터는 DB가 아닌 Redis 캐시로 처리.
**트레이드오프**: SQLite↔PostgreSQL 미묘한 차이(동시성, 타입) 존재. SQLAlchemy + 마이그레이션으로 관리.

### ADR-005: LLM은 최종 게이트, 알고리즘이 1차 필터
> ⚠️ **제공자 부분은 ADR-007로 대체됨**: 본문의 "Claude(claude-sonnet-4-6)"는 OpenAI로 교체됐다.
> 단, "LLM은 최종 판단만, 알고리즘이 1차 필터" 원칙은 그대로 유효하다.

**결정**: 알고리즘 3레이어를 모두 통과한 종목만 Claude 판단 에이전트(claude-sonnet-4-6)에 보낸다.
**이유**: 모든 종목을 Claude에 보내면 비용·지연·환각 리스크가 크다. 결정론적 규칙으로 후보를 좁히고, Claude는 뉴스·맥락 종합 최종 판단에만 사용한다.
**트레이드오프**: 알고리즘이 놓친 기회는 Claude까지 가지 못한다. 규칙 튜닝으로 보완.

### ADR-006: SDD → TDD 강제 (스펙 우선, 테스트 우선)
**결정**: 기능마다 `specs/기능명.md`(입력/출력/엣지케이스) → `tests/test_기능명.py`(Red) → 구현(Green) → 리팩터 순서를 강제한다.
**이유**: 금융 로직은 엣지케이스(부분 체결, 슬리피지, 0 수량, 음수 잔고)에서 사고가 난다. 스펙·테스트를 먼저 박아 회귀를 막는다.
**트레이드오프**: 초기 속도 저하. 그러나 실거래 시스템에서 회귀 비용이 훨씬 크다.

### ADR-007: LLM 제공자는 OpenAI (Claude 대체)
**결정**: 봇의 LLM 기능(판단 에이전트·목표플랜 근거·뉴스 센티먼트·시황 방향성·리포트 코멘트)은 **OpenAI API**로 구현한다. 모델은 `OPENAI_MODEL` 환경변수(기본 `gpt-4o`), 키는 `OPENAI_API_KEY`(.env). 코드의 `Claude*Provider` 골격은 `OpenAI*Provider`로 마이그레이션한다.
**이유**: 초기 설계는 Claude(claude-sonnet-4-6) 기준이었으나, 운영 주체가 보유한 자격증명이 OpenAI 키다. provider 추상화(Mock/실제) 덕분에 LLM 제공자 교체 비용이 낮다. ADR-005의 "LLM은 최종 판단/근거만, 알고리즘이 1차 필터·안전 수치 단일 진실"은 그대로 유지(제공자만 OpenAI로 교체).
**트레이드오프**: 문서·코드의 "Claude" 네이밍을 OpenAI로 정리하는 마이그레이션 비용. 안전 불변식(ADR-003: 하드캡, LLM이 리스크 수치 못 바꿈)은 제공자와 무관하게 유지된다.
**불변**: 키 부재 시 모든 LLM provider는 Mock으로 안전 fallback하며, 테스트는 네트워크 없이 mock 클라이언트로 검증한다.

### ADR-008: 기술지표는 TA-lib 없이 pandas/numpy로 직접 계산
**결정**: EMA/RSI/MACD/ATR/거래량 등 모든 지표를 `algorithms/`에서 pandas/numpy로 직접 구현한다. TA-lib에 의존하지 않는다.
**이유**: 초기 설계(ADR-001 본문)는 TA-lib 생태계를 가정했으나, TA-lib는 C 라이브러리 네이티브 빌드가 필요해 환경 의존성·설치 실패 리스크가 크다. 지표 구현은 단순하고, 순수 함수(ADR-002)로 직접 계산하면 결정론·테스트 용이성을 유지하면서 빌드 의존성을 제거할 수 있다.
**트레이드오프**: 지표를 직접 구현·검증하는 비용. 그러나 환경 재현성과 CI 안정성 이득이 크다. (ADR-001 본문의 "TA-lib" 언급은 이 ADR로 정정된다.)

### ADR-009: 리스크 한도 단위는 분수로 통일, 매매당 vs 포트폴리오 리스크 분리
**결정**: `RiskLimits`의 모든 한도(`max_risk_pct`/`max_drawdown_pct`/`max_position_pct`/`max_portfolio_loss_pct`)는 분수(0.05 = 5%)로 통일한다. 또한 `max_risk_pct`("1회 매매당" 허용 손실, sizing 전용)와 `max_portfolio_loss_pct`("포트폴리오 전체" 미실현 손실 정지선, RiskAgent 전용)를 **의미가 다른 별도 필드**로 분리한다.
**이유**: 단위 불일치(퍼센트 vs 분수)와 두 개념의 혼용은 사이징·정지선 계산에서 한도를 의도치 않게 100배 오해석하는 치명적 버그를 낳는다. 단일 단위(분수)로 sizing·RiskAgent·goal_planner가 일관 소비하게 하고, 매매당 리스크와 계좌 단위 정지선을 분리해 각 안전장치가 올바른 임계값을 쓰게 한다.
**트레이드오프**: 기존 필드 의미 변경에 따른 마이그레이션·테스트 갱신 비용. 그러나 실거래 안전 불변식(ADR-003)의 정확성이 우선이다.

### ADR-010: 사이징은 콜드스타트 shrinkage + MDD governor + 레짐 배수 (fractional Kelly 라벨버그 수정)
**결정**: `kelly_fraction`을 진짜 fractional Kelly로 수정한다 — `f_used = clamp(fraction × max(0, f_full), 0, cap)`로 `fraction`(비례축소)과 `cap`(절대 상한)을 분리한다(기존 `min(f, cap)`은 cap 미만 베팅을 축소하지 않는 라벨버그였다). 거래기록이 없는 콜드스타트는 `effective_kelly_fraction`의 shrinkage(`w = n/(n+k)`, `f_eff = w·kelly + (1−w)·prior`)로 prior(기본 0.0 = 켈리 미사용)에서 표본 누적에 따라 경험적 켈리로 점진 전환한다. 레짐 사이징 배수(헌장 §8)는 `regime_adjusted_fraction`이라는 켈리 위 **별도 레이어**로 곱한다(A×1.0/B×0.5/C·D×0).
**이유**: (1) fractional Kelly의 핵심은 *모든* 베팅의 비례축소인데 `min(f,cap)`은 이를 깨뜨려 작은 베팅에서 과대 사이징을 냈다. (2) 콜드스타트에서 경험적 켈리(표본 0)를 신뢰하면 과대 베팅→파산이라, 표본이 쌓이기 전엔 보수적 고정 비율에 위임해야 한다. (3) `fraction`은 추상값이 아니라 백테스트 MDD가 설계목표 ≤15%(헌장 §6)로 나오도록 역튜닝되는 governor다. (4) 레짐 배수를 켈리 함수에 섞지 않고 분리해야 켈리 순수성과 백테스트 A/B 튜닝이 가능하다.
**불변(ADR-003)**: 어떤 경로(fractional·shrinkage·레짐배수·appetite)로도 `position_size`의 최종 risk_amount는 `equity × max_risk_pct`를 초과하지 못한다. 레짐 배수(≤1.0)는 캡을 *올리지* 못하고 축소만 하며, `position_size`의 max_qty 하드클램프가 2중 안전으로 남는다. 켈리 입력(win_rate/win_loss_ratio/sample_size) 출처는 백테스트 엔진(1순위)→실거래 로그(2순위)이며, 백테스트 미구현 현재는 prior=0 콜드스타트 경로만 활성이다.
**트레이드오프**: 기존 켈리 테스트 기대값 변경(0.25→0.20 등)과 신규 레이어 추가 비용. 그러나 헌장 §6/§7의 MDD governor 원칙과 안전이 우선이다. (참고: `tasks/kelly-fix-prompt.md`)

### ADR-011: v1 백테스트 엔진은 일봉·미래참조 차단·보수적 비용·LLM 격리·생존편향 경고
**결정**: `algorithms/backtest.py`는 step0~4 순수 함수를 오케스트레이션하는 **결정론적 순수 함수** 엔진이다. (1) **v1=일봉 전용** 완전 전략(진입+청산+사이징+레짐+비용)으로 go/no-go 게이트를 만든다(1시간봉 v2는 v1 통과 후). (2) **미래참조 차단**: 바 t 의사결정은 `[:t+1]`만 보고 신호는 일봉 종가, **체결은 다음날 시가**(t+1). (3) **비용은 보수적**(slippage_bps 기본 5, 0 금지 — Robinhood 무수수료여도 PFOF 슬리피지 반영). (4) **LLM 격리**: 백테스트는 결정론적 가격 엣지만 측정 — 센티먼트/판단 provider는 주입형 중립 mock, 실호출 금지(비결정성·미래참조). (5) **워크포워드** train/test split로 OOS 열화 측정(헌장 §10②). (6) `BacktestResult`에 Sharpe(주지표)·Sortino·MDD·SPY 벤치마크 비교·레짐 분해·켈리 입력(win_rate/win_loss_ratio/total_trades)을 담고, **생존편향 경고를 항상 명기**한다.
**이유**: 거래기록도 백테스트도 없어 엣지·켈리 입력을 알 수 없다 — 엔진이 헌장 전략의 엣지를 과거에 재생해 측정하고 그 산출을 켈리 콜드스타트로 공급한다(ADR-010 연결). 미래참조는 백테스트를 거짓말하게 만드는 1순위 오류라 종가→다음날 시가 체결로 원천 차단한다. 일봉은 싸고 깨끗해 분봉 인프라 없이 fail-fast가 가능하다(헌장 §10). 데이터 로딩(I/O)은 엔진에 넣지 않고 입력 DataFrame만 소비한다(ADR-002 순수성, 로딩은 step6/agents).
**불변·경고**: v1 숫자는 현재 유니버스+무료데이터라 **생존편향 내장 → "낙관적 상한"**이며 **라이브 greenlight가 아니다**(헌장 §3). 라이브 전 생존편향 없는 벤더로 point-in-time 재검증이 필수다. 0거래·무변동도 0분모 폭발 없이 안전(win_rate=0, sharpe=0). (참고: `tasks/backtest-engine-prompt.md`)
