# Architecture Decision Records

## 철학
실거래로 실제 돈이 오가는 시스템이다. **안전(리스크 차단)과 데이터 무결성이 속도보다 우선**한다.
모든 자동 주문은 알고리즘(규칙) → Claude(판단) → 리스크 게이트(차단)의 다단계 승인을 통과해야 한다.
SDD → TDD를 강제해 매매 로직의 회귀를 막는다. 외부 의존성은 backend에 격리해 프론트가 직접 거래소/AI를 호출하지 못하게 한다.

---

### ADR-001: 프론트/백 분리 (Next.js + FastAPI)
**결정**: 프론트엔드는 Next.js 14, 백엔드는 별도 FastAPI 서버로 분리한다.
**이유**: 알고리즘·에이전트가 Python(Pandas, TA-lib) 생태계에 묶여 있고, Robinhood MCP·Claude 호출을 한 곳(backend)에 격리해야 보안·리스크 통제가 가능하다. 실시간 push는 FastAPI WebSocket으로 처리.
**트레이드오프**: 단일 Next.js 풀스택 대비 배포 단위·타입 동기화 비용 증가. 타입은 수동 동기화로 감수.

### ADR-002: 알고리즘은 순수 함수, 에이전트는 상태 루프
**결정**: `algorithms/`는 부수효과 없는 순수 함수(입력 DataFrame→출력 시그널/사이즈), `agents/`는 주기 실행 루프로 분리한다.
**이유**: 매매 핵심 로직(시그널/사이징)을 외부 의존성 없이 단위 테스트(TDD)할 수 있어야 한다. I/O(MCP, Claude, DB)는 에이전트 레이어로 밀어낸다.
**트레이드오프**: 레이어 경계 유지 비용. 그러나 테스트 용이성과 회귀 방지 이득이 크다.

### ADR-003: 리스크 에이전트 kill-switch는 PreToolUse hook으로 강제
**결정**: 주문 실행 툴 호출 전 PreToolUse hook이 리스크 에이전트를 강제 호출하고, 한도 초과 시 주문을 차단한다.
**이유**: 코드 경로 어디서 주문이 나가든 단일 게이트를 통과하게 만들어, 버그·우회로 인한 한도 초과 주문을 원천 차단한다. 안전이 최우선.
**트레이드오프**: 모든 주문에 리스크 체크 지연이 추가된다. 안전을 위해 감수한다.

### ADR-004: 개발 DB는 SQLite, 캐시/실시간은 Redis
**결정**: 개발은 SQLite로 시작하고 프로덕션에서 PostgreSQL로 전환한다. 실시간 가격·에이전트 공유 상태는 Redis.
**이유**: 1인 개발 초기 속도를 위해 SQLite. SQLAlchemy로 추상화해 전환 비용을 낮춘다. 1초 갱신 실시간 데이터는 DB가 아닌 Redis 캐시로 처리.
**트레이드오프**: SQLite↔PostgreSQL 미묘한 차이(동시성, 타입) 존재. SQLAlchemy + 마이그레이션으로 관리.

### ADR-005: Claude는 최종 게이트, 알고리즘이 1차 필터
**결정**: 알고리즘 3레이어를 모두 통과한 종목만 Claude 판단 에이전트(claude-sonnet-4-6)에 보낸다.
**이유**: 모든 종목을 Claude에 보내면 비용·지연·환각 리스크가 크다. 결정론적 규칙으로 후보를 좁히고, Claude는 뉴스·맥락 종합 최종 판단에만 사용한다.
**트레이드오프**: 알고리즘이 놓친 기회는 Claude까지 가지 못한다. 규칙 튜닝으로 보완.

### ADR-006: SDD → TDD 강제 (스펙 우선, 테스트 우선)
**결정**: 기능마다 `specs/기능명.md`(입력/출력/엣지케이스) → `tests/test_기능명.py`(Red) → 구현(Green) → 리팩터 순서를 강제한다.
**이유**: 금융 로직은 엣지케이스(부분 체결, 슬리피지, 0 수량, 음수 잔고)에서 사고가 난다. 스펙·테스트를 먼저 박아 회귀를 막는다.
**트레이드오프**: 초기 속도 저하. 그러나 실거래 시스템에서 회귀 비용이 훨씬 크다.
