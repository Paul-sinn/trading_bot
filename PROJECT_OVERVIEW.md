# Custom Trading Bot — 프로젝트 설명서 (심사용)

> 이 문서는 프로젝트를 처음 보는 심사관이 **무엇을 만들었는지, 어떻게 돌아가는지, 데이터가 어떻게 흐르는지**를
> 한 번에 파악하도록 작성했습니다. 세부 설계는 `docs/PRD.md`, `docs/ARCHITECTURE.md`, `docs/ADR.md` 참조.

---

## 1. 한 줄 요약

**알고리즘 시그널(규칙) + LLM 최종 판단을 결합해, 개인 투자자의 주식 계좌에서 "리스크 한도 안에서만" 자동매매하고 실시간으로 모니터링하는 봇.**

핵심 가치: **안전(리스크 차단)과 데이터 무결성이 속도보다 우선.** 실제 돈이 오가므로, 모든 자동 주문은 다단계 승인(규칙 → AI → 리스크 게이트)을 통과해야 한다.

---

## 2. 전체 구조 — 폴리글랏 모노레포

단일 앱이 아니라, 책임별로 4개 레이어로 분리된 모노레포다.

```
trading-bot/
├── frontend/     # Next.js 14 — 사용자 화면 6개 (대시보드/거래기록/AI분석/목표/투자성향)
├── backend/      # FastAPI — 모든 외부 API·DB·시크릿의 단일 관문(SSOT)
├── agents/       # 서브에이전트 6개 — 스캐너/판단/실행/리스크/리포트/알림 (각자 주기로 도는 루프)
├── algorithms/   # 매매 두뇌 — 시그널/필터/사이징 (부수효과 없는 순수 함수)
├── specs/        # 기능별 SDD 스펙 문서 (구현 전에 입출력·엣지케이스 정의)
├── tests/        # 기능별 TDD 테스트
├── phases/       # 단계별 구현 계획 (Harness 자동 실행용)
└── docs/         # PRD / ARCHITECTURE / ADR / UI_GUIDE
```

### 레이어 책임 분리 (왜 이렇게 나눴나)

| 레이어 | 책임 | 핵심 규칙 |
|--------|------|-----------|
| `algorithms/` | 매매 계산(지표·수량·리스크) | **순수 함수** — I/O·전역상태 없음 → 결정론적이라 테스트가 쉽고 회귀를 막는다 |
| `agents/` | 주기 실행 루프 + I/O 조율 | 외부 호출(거래소·LLM·DB)은 여기로 격리 |
| `backend/` | 외부 API·DB·시크릿의 **단일 관문** | 프론트는 거래소/LLM을 절대 직접 호출하지 못한다 |
| `frontend/` | 화면 표시 + 사용자 입력 | backend의 REST/WebSocket만 호출 |

> **단일 진실 공급원(SSOT) = backend.** 보안·리스크 통제를 한 곳에 모으기 위함. (ADR-001)

## 3. 매매 알고리즘 — 3레이어 (핵심)

종목 하나가 매수까지 가려면 **3개 레이어를 순서대로 모두 통과**해야 한다.

```
워치리스트 종목
  │
  ▼ Layer 1 · 시그널 생성  (algorithms/signals.py)
  │    EMA 크로스(9/21) · RSI(14) · MACD 히스토그램 → 각각 BULLISH/NEUTRAL/BEARISH → 다수결
  │
  ▼ Layer 2 · 필터링  (algorithms/filters.py)   ※ 4개 전부 AND
  │    거래량 급등 · ATR 변동성(≤8%) · 뉴스 센티먼트(LLM) · VIX(≤30)
  │
  │  ── 여기까지 통과한 종목만 "후보" ──
  │
  ▼ LLM 판단  (agents/decision.py)
  │    뉴스+차트 맥락 종합 → 매수(BUY) / 보유(HOLD) / 매도(SELL)
  │
  ▼ Layer 3 · 포지션 사이징  (algorithms/sizing.py)
  │    Kelly 변형(상한 0.25) · 스탑로스 = 진입가-(ATR×배수) · 투자성향 가중 · 최대 리스크% 한도
  │    → 진입 수량 + 스탑가 결정 (리스크 한도 절대 초과 불가)
  │
  ▼ 리스크 게이트  (agents/risk.py)  → 통과해야 주문
  ▼ 실행 → 체결 → 리포트 → 알림
```

### Layer별 요점

**Layer 1 — 시그널 (추세 판단)**
- EMA 9/21 골든/데드 크로스, RSI 과매수(70)/과매도(30), MACD 히스토그램 0선 전환.
- 3개 지표를 **다수결**로 종합. 데이터 부족하면 NEUTRAL(안전).
- TA-Lib 대신 pandas/numpy로 직접 계산(환경 의존성 제거).

**Layer 2 — 필터 (잡음·위험 제거, 전부 AND)**
- 거래량이 평소의 1.5배 이상 터졌는가 / 변동성이 과하지 않은가 / 뉴스가 부정적이지 않은가 / 시장 공포(VIX)가 높지 않은가.
- 하나라도 실패하면 후보에서 탈락.

**Layer 3 — 사이징 (얼마나 살까)**
- **Kelly Criterion 변형**으로 베팅 비율 결정(풀켈리 파산 방지 위해 상한 0.25 = half-Kelly).
- 스탑로스를 ATR(변동성) 기반으로 설정.
- 투자성향 슬라이더(보수↔공격)를 가중치로 반영.
- **🔒 하드캡: 어떤 입력이든 1회 손실액이 `계좌자산 × 최대리스크%`를 절대 초과하지 못한다.** (ADR-003 — 시스템의 가장 중요한 안전장치)

## 4. 서브에이전트 6개

각 에이전트는 공통 인터페이스(`agents/base.py`: start/stop/tick)를 구현하고, 자기 주기로 도는 독립 루프다. 모두 `AgentRegistry`에 등록되어 **리스크 에이전트의 kill-switch**로 일괄 정지될 수 있다.

| 에이전트 | 파일 | 역할 |
|----------|------|------|
| **스캐너** | `agents/scanner.py` | 워치리스트 순환 스캔(1분 주기) → Layer1+2 적용 → 후보 리스트업 |
| **판단** | `agents/decision.py` | LLM 호출, 뉴스+차트 종합 → BUY/HOLD/SELL (불확실 시 HOLD) |
| **실행** | `agents/executor.py` | 리스크 게이트 통과 후 주문 실행, 체결 확인, 슬리피지 기록 |
| **리스크** | `agents/risk.py` | 실시간 리스크 계산, 한도 초과 시 **전 에이전트 강제 정지(kill-switch)** |
| **리포트** | `agents/reporter.py` | 일간/주간 성과 집계, AI 코멘트, DB 저장 |
| **알림** | `agents/notifier.py` | 체결·리스크·목표달성 이벤트 → 슬랙/SMS (채널 실패 격리) |

## 5. 데이터 흐름 (5가지 시나리오)

```
[① 자동매매 루프]
스캐너(1분) → 3레이어 → 판단(LLM) → [리스크 게이트] → 실행(주문) → 체결 → DB → WebSocket → UI

[② 실시간 동기화]
Robinhood(MCP) → backend(캐시) → WebSocket → frontend (1초 갱신: 총자산/손익/티커)

[③ 리스크 차단 (kill-switch)]
리스크 에이전트(실시간 손실% 계산) → 한도 초과 → 전 에이전트 정지 → 알림 발송

[④ 수동 거래]
UI 티커 검색 → 수량 입력 → backend → [리스크 게이트] → 주문 → 체결 → UI

[⑤ 목표 기반 AI 개인화] (아래 7장)
목표금액+기간 입력 → AI 분석 → 역산된 리스크·투자성향 세팅 → 검토 후 적용 → 매매에 반영
```

## 6. 안전 설계 — 3중 방어 (가장 중요)

실거래 시스템이므로, 사고를 막는 다층 방어를 코드 레벨에서 강제한다.

1. **다단계 승인** — 자동 주문은 반드시 `규칙(3레이어) → LLM 판단 → 리스크 게이트` 순서를 통과. 이 경로를 우회하는 주문 코드는 금지.
2. **리스크 하드캡** — 포지션 사이징이 계산하는 1회 손실액은 수학적으로 `자산 × 최대리스크%`를 초과할 수 없다(수량을 깎아서라도). 목표가 아무리 공격적이어도, 시스템 절대 한도(`SYSTEM_MAX_RISK_PCT`)를 넘지 못한다.
3. **kill-switch** — 리스크 에이전트가 한도 초과를 감지하면 모든 에이전트를 즉시 정지. 주문 실행 직전 **PreToolUse hook**이 리스크 체크를 강제하며, 판단이 불확실하면 **차단(fail-closed)**.

> 추가로: **LLM은 "설명·판단"만 한다.** 리스크 수치·포지션 크기 같은 안전 수치는 알고리즘이 단일 진실이며, LLM(AI 환각 포함)이 이 수치를 덮어쓰지 못하게 코드로 막는다. (ADR-003/005)

## 7. 목표 기반 AI 개인화 ("AI 분석하기")

이 프로젝트의 차별점. 사용자가 **목표금액 + 목표기간(개월)** 을 정하면, AI가 그 목표 달성에 필요한 리스크·투자성향 세팅을 **역산해서 개인화**해주고, 그 세팅으로 매매한다.

```
[목표 & 리스크 탭]
사용자: 목표금액 + 목표기간 + 모드(안전[추천]/공격) 입력 → "AI 분석하기"
   │
   ▼ backend POST /api/goal-plan
   ├─ ⓐ 결정론 역산 (algorithms/goal_planner.py)
   │    목표금액/기간 → 필요 월수익률 → 실현가능성(현실적/도전적/비현실적)
   │    → 투자성향(appetite)·최대리스크%·드로우다운·포지션·스탑배수 역산
   │    🔒 어떤 목표든 max_risk_pct는 시스템 하드캡 이하로 강제
   └─ ⓑ AI 근거 생성 (LLM) — 수치는 못 바꾸고 "왜 이런 세팅인지" 설명만
   │
   ▼ 화면에 역산 세팅 + 실현가능성 배지 + AI 근거 표시
   ▼ 사용자가 검토 후 "적용" → 활성 세팅으로 저장 → 매매 루프(사이징/리스크)에 반영
```

**설계 원칙:**
- **"분석" ≠ "적용"**: 분석은 시스템을 아무것도 바꾸지 않는다. "적용"을 눌러야 반영(검토 후 적용).
- **숫자는 알고리즘, AI는 설명**: 비현실적 목표(예: 1주일에 2배)를 넣어도 risk%는 하드캡에서 멈추고 "비현실적"이라고 경고한다.

## 10. 개발 방법론 — SDD → TDD + Harness

품질을 강제하기 위한 워크플로우.

- **SDD → TDD 강제**: 기능마다 `specs/기능명.md`(스펙) → `tests/test_기능명.py`(실패 테스트) → 구현(통과) → 리팩터 순서. 금융 로직은 엣지케이스에서 사고가 나므로 스펙·테스트를 먼저 박는다.
- **Harness 자동 실행**: 큰 작업을 `phases/{task}/step{N}.md`로 잘게 나누고, 스크립트가 각 step을 독립 세션으로 자동 구현·검증·커밋. 매 step에 프로젝트 규칙(CLAUDE.md+docs)을 가드레일로 주입.
- **git hooks**: pre-commit(린트), pre-push(전체 테스트 통과해야 push) — 깨진 코드가 올라가지 못하게 막는다.

| Phase | 내용 | 상태 |
|-------|------|------|
| 0 · foundation | 폴더구조, hooks, FastAPI+WS, 알고리즘 3레이어 | ✅ 완료 |
| 1 · agents | 서브에이전트 6개 | ✅ 완료 |
| 2 · frontend | Next.js 6개 페이지 | ✅ 완료 |
| 3 · goal-planner | 목표 기반 AI 개인화 | ✅ 완료 |
| 4 · integration | OpenAI 실연동 + 적용세팅→매매루프 연결 + E2E | 🔜 준비됨(미실행) |

**테스트 규모:** Python 220+ 테스트 + Frontend 28 테스트 (각 기능 SDD→TDD).

**아직 mock인 부분 (의도된 격리):**
- **LLM(OpenAI)**: provider 골격까지. 실제 호출은 Phase 4에서 연동(키는 `.env`에 사용자가 주입). 키 없으면 안전하게 Mock으로 fallback.
- **Robinhood**: MockProvider 기본값(실거래/실조회 없음). 실제 MCP 연동은 안전 토글(`ROBINHOOD_MCP_ENABLED`)로 켜야 동작.
- → 즉, **지금은 실거래가 절대 나가지 않는 안전 상태**에서 전체 파이프라인이 검증되어 있다.

**진행 중인 보강 작업:**
- 매매 알고리즘 안전 점검 중 **리스크 한도 단위 불일치 버그**(퍼센트 vs 분수)와 **의미 혼용**(매매당 리스크 ↔ 포트폴리오 정지선)을 발견 → 단위를 분수로 통일하고 두 개념을 별도 필드로 분리하는 수정 진행 중. (Phase 4 통합 전 선반영)
